# ADR-0002 — sqlc over an ORM

**Status:** Accepted (Phase 1)
**Date:** 2026-05-03

## Context

Section 9 of the spec mandates "all queries via sqlc — zero raw string
interpolation permitted." We considered `gorm`, `ent`, and hand-rolled
`pgx` with helpers. Multi-tenant isolation by `org_id` makes review-time
verification of scoping clauses critical; an ORM hides that scoping
behind builder chains.

## Decision

Use **sqlc 1.28** to generate type-safe Go from `.sql` files committed
alongside migrations. Migrations live in `internal/db/migrations/` and
are managed by `golang-migrate`. Generated code lands in
`internal/db/sqlc/` (gitignored — regenerated by `make sqlc`).

Conventions:

- Every tenant-scoped query takes `org_id` as the first parameter and
  has it in the `WHERE` clause. CI greps for queries on tenant tables
  that lack `org_id` and fails the build.
- `sqlc` config uses `emit_interface: true` so the application layer
  depends on a `Querier` interface, not the generated struct.
- Transactions are composed in the application layer via
  `db.WithTx(ctx, func(q Querier) error { ... })`.

## Consequences

**Positive.** Compile-time safety on column names and types. Reviewers
read SQL, not builder chains. Tenant scoping is visually obvious.

**Negative.** Schema changes are a two-step dance (migration + `make
sqlc`). Mitigated by a `make migrate-up && make sqlc` shortcut and a CI
check that fails if generated code is out of sync with `*.sql` files.
